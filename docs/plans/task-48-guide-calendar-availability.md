# Таск 48 — календарь доступности гида

Дата: 2026-07-08 21:07 +04
Обновлено: 2026-07-08

## Контекст

Бадма предложил дополнить недавно внедрённую кнопку активности/деактивации гида моделью Tripster: гид заранее отмечает даты и временные окна, когда он недоступен, и в эти периоды турист не может забронировать готовую экскурсию или отправить именной запрос этому гиду.

Календарь Tripster был осмотрен через авторизованную сессию. Проверенные наблюдения (без реквизитов и скриншотов):

- в кабинете гида есть отдельная вкладка **«Календарь»**;
- виден дневной таймлайн с шагом **30 минут**;
- справа — мини-календарь месяца с индикаторами занятых дней;
- есть фильтр **«Что показывать» / «Все предложения»**;
- есть кнопки-действия **«Закрыть время»** и **«Закрыть день»**.

Вывод: Tripster использует **календарь недоступности как отдельный слой**, а не как замену общему переключателю активности. Именно эту разницу мы и фиксируем в модели Провoдника.

---

## 1. Продуктовая модель: три уровня доступности

Ключевое решение — не смешивать три разных механизма. У них разные владельцы, разные причины и разный приоритет.

| Уровень | Кто управляет | Смысл | Приоритет при расчёте |
|---|---|---|---|
| **A. Глобальная активность гида** (`is_available`) | сам гид | «Главный рубильник»: я работаю / временно не принимаю ничего нового | Высокий — выключает всё |
| **B. Календарь недоступности** (`guide_availability_blocks`) | сам гид | Точечные даты/окна, когда я занят: отпуск, не сезон, жара, основная работа | Средний — блокирует только пересекающиеся даты/слоты |
| **C. Блокировка платформой / поддержкой** (модерация, suspension) | админ/саппорт | Аккаунт скрыт по решению платформы (жалобы, проверка, нарушение) | Наивысший — перекрывает A и B, гид сам снять не может |

Формула итоговой доступности:

```text
гид доступен на (дата, [время]) =
      НЕТ блокировки платформы (C)
  И   глобальная активность включена (A: is_available = true, аккаунт активен)
  И   запрошенный интервал НЕ пересекается ни с одним активным блоком календаря (B)
```

Порядок проверки важен: сначала C, потом A, потом B. Это влияет и на текст ошибки (см. раздел о резолюции), и на то, кто может снять ограничение.

Связанный ландмайн: видимость гида в каталоге уже зависит от `is_available` + активности аккаунта. Календарь (B) не должен ломать эту логику; он добавляется поверх, а не вместо.

## 2. Решение: почему кнопка активности остаётся

Кнопка (`is_available`) **сохраняется** и после появления календаря. Причины:

- **Разные сценарии.** Кнопка — аварийный режим «выключить себя целиком одним кликом» (заболел, уехал, перегружен). Календарь — плановый режим «закрыть конкретные дни/часы заранее» (отпуск, сезон, семейные дни, основная работа по часам).
- **Разная ментальная модель.** Гид не должен «набивать» календарь на полгода вперёд, чтобы просто выключиться сегодня. И наоборот — не должен выключать весь профиль, чтобы закрыть одну субботу.
- **Разная обратимость.** Кнопка — быстро туда-обратно. Календарь — расписание, которое живёт долго и редактируется по частям.

Что кнопка означает **после** появления календаря:

- **Активен** — гид работает по календарю: недоступны только явно закрытые дни/окна.
- **Неактивен** — гид полностью недоступен для новых бронирований и именных запросов, **независимо** от календаря. Календарь при этом не стирается — при повторном включении расписание остаётся прежним.

Для гида это подаётся как один раздел **«Моя доступность»**, а не две отдельные функции:

```text
Сейчас я: Активен / Неактивен      ← рубильник (A)
Календарь: закрытые даты и часы     ← расписание исключений (B)
```

## 3. Поведение для туриста

### 3.1 Готовые экскурсии

Если турист выбирает дату/время, попадающее в закрытый период гида:

- слот показывается как **недоступный** (не кликабельный / визуально погашен);
- забронировать у этого гида нельзя;
- если экскурсию ведут несколько гидов/форматов — предлагаем доступных;
- если экскурсия только у этого гида — нейтральный текст: **«Гид недоступен в выбранное время»**, без причины и без деталей календаря гида.

### 3.2 Именные запросы конкретному гиду

Если турист отправляет запрос конкретному гиду и дата/время попадает в закрытый период:

- запрос **не уходит** как обычная активная заявка;
- предупреждение показывается **до отправки**;
- предлагаемые варианты:
  - выбрать другую дату/время;
  - отправить запрос без жёсткой даты (если такой сценарий допускаем — см. открытые вопросы);
  - выбрать другого гида.

**Важно:** блокировка обязана работать не только в UI, но и на сервере — запрос нельзя создать в обход интерфейса (прямой вызов API). Единый источник правды — серверная функция резолюции (раздел 8).

Приватность: туристу никогда не раскрываем причину блока (отпуск/работа и т.п.) и содержимое чужого календаря — только факт «недоступно».

## 4. UX гида и принципы формулировок

### 4.1 Экран «Календарь» / «Моя доступность»

- сверху: текущий общий статус + кнопка активности (A);
- центр: календарь месяца (мини-календарь) + дневной таймлайн с шагом 30 минут (модель Tripster);
- действия: **«Закрыть день»** и **«Закрыть время»** (окно в конкретный день);
- ниже: список **«Закрытые периоды»** с возможностью редактировать/удалить.

### 4.2 Действия MVP (точный список)

1. **Закрыть весь день.**
2. **Закрыть диапазон дат** (например, отпуск с… по…).
3. **Закрыть временное окно** в конкретный день (напр. 14:00–18:00).
4. **Указать причину для себя** (не видна туристу): отпуск, не сезон, жара, основная работа, другое.
5. **Изменить/удалить** закрытый период.
6. **Видеть** закрытые периоды на месяц/год вперёд.

### 4.3 Принципы безопасных формулировок (copy)

- Гиду — прямо и по делу: «Закрыть день», «Закрыть время», «Этот период закрыт для новых заявок».
- Туристу — нейтрально и без причин: «Гид недоступен в выбранное время».
- Не обещать того, что не гарантируем: закрытие **не отменяет** уже принятые договорённости — так и писать.
- Никаких персональных причин гида в тексте для туриста.

## 5. Админ/поддержка и аудит

### 5.1 Что видит и может саппорт

- Просмотр календаря гида и его закрытых периодов (для разбора спорных ситуаций).
- Блокировка платформы (уровень C) — отдельный механизм, не пересекается с гидовым календарём; гид её сам снять не может.
- В спорных кейсах (гид закрыл дату, где уже есть подтверждённая бронь) — ручное решение поддержки.

### 5.2 Аудит

Каждое изменение доступности должно быть прослеживаемым:

- кто создал/изменил/удалил блок (`created_by`, признак «гид» vs «саппорт/система»);
- когда (`created_at`, `updated_at`, `deleted_at` — мягкое удаление);
- источник (`source`: manual / calendar / system);
- смена глобальной активности (A) и платформенной блокировки (C) логируется отдельно от блоков календаря.

Цель аудита: восстановить историю «почему в этот момент гид был недоступен» без догадок.

## 6. Модель данных (уровень плана)

Новая сущность: **`guide_availability_blocks`**.

| Поле | Тип (ориентир) | Назначение |
|---|---|---|
| `id` | uuid | PK |
| `guide_id` | uuid → guide | владелец блока |
| `start_at` | timestamptz | начало интервала (UTC в БД) |
| `end_at` | timestamptz | конец интервала |
| `all_day` | bool | флаг «весь день», упрощает date-only логику |
| `reason` | text/enum | причина для гида (не для туриста) |
| `source` | enum | manual / calendar / system |
| `created_by` | uuid | автор (гид или саппорт) |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | |
| `deleted_at` | timestamptz null | мягкое удаление для аудита |

Повторяющиеся правила (еженедельно, выходные, сезон) — **отдельная сущность `guide_availability_rules`, не в MVP.** Для MVP храним только конкретные интервалы.

### 6.1 Безопасность и RLS (принципы, не финальный SQL)

- **Владение:** гид может читать/писать только свои блоки (`guide_id = auth-гид`). Запись чужих блоков запрещена на уровне RLS.
- **Чтение туристом:** турист **не** читает таблицу блоков напрямую. Доступность отдаётся только как булев результат серверной функции резолюции — приватность причин сохраняется.
- **Саппорт/сервис:** пишет через сервисную роль с явным `created_by` и `source = system`.
- **Часовые пояса:** хранить в UTC (`timestamptz`), считать доступность в таймзоне тура/гида; учитывать граничные случаи полуночи и `all_day`.
- **Целостность:** проверка `end_at > start_at`; для `all_day` окно нормализуется к границам суток.

## 7. Правила резолюции доступности

Единая логика «доступен ли гид на запрошенный интервал». Порядок: C → A → B (раздел 1).

- **Запрос только с датой (без времени).**
  - закрыт весь день (`all_day` или блок покрывает сутки целиком) → **недоступно**;
  - закрыта часть дня → **предупредить и попросить выбрать время**; без времени именной запрос не уходит как «доступный»;
  - день полностью свободен → доступно.
- **Запрос с временным окном.** Доступно, только если интервал **не пересекается** ни с одним активным блоком. Любое пересечение (даже частичное) → недоступно на этот слот.
- **Пересекающиеся/дублирующиеся блоки.** Трактуем как объединение (union): достаточно одного пересечения, чтобы слот был закрыт. Наложение блоков — не ошибка.
- **Блок «весь день».** Перекрывает любые запросы этой даты, включая date-only и любые окна.
- **Уже существующие бронирования.**
  - при попытке гида закрыть дату с **подтверждённой** бронью — предупреждение: «На эту дату уже есть заявка/бронь», «Закрытие не отменяет существующие договорённости», «Новые заявки на это время будут заблокированы»;
  - жёсткий вариант (обсуждаемо): запрет закрывать даты с подтверждёнными бронями без ручного решения поддержки;
  - закрытие даты **никогда не удаляет** и не отменяет уже принятые заявки — старые статусы остаются как есть, чтобы не рвать коммуникацию.

## 8. Интеграция (API/сервис) и критерии приёмки

### 8.1 Единая точка проверки

Единая серверная функция/сервис:

```text
isGuideAvailable(guideId, dateTimeRange, context) -> { available, reason? }
```

Использовать **везде**, где гид может быть назначен:

- создание бронирования готовой экскурсии;
- создание именного запроса гиду;
- показ доступных дат/слотов туристу;
- админка/модерация, если оттуда создаются заявки;
- любые API-ручки, которые способны назначить гида.

Нельзя оставлять ни одной ветки, создающей заявку в обход этой проверки: при реализации нужно проверить все места, где создаётся бронирование, именной запрос или назначение гида.

### 8.2 Критерии приёмки

- [ ] Гид может закрыть день, диапазон дат и окно в дне; изменить и удалить их.
- [ ] Турист не может забронировать/отправить именной запрос на закрытый интервал — **ни через UI, ни напрямую через API**.
- [ ] Date-only запрос при частично закрытом дне требует выбора времени.
- [ ] Пересекающиеся блоки трактуются как union, `all_day` перекрывает всё.
- [ ] Существующие подтверждённые брони не отменяются закрытием даты; гид получает предупреждение.
- [ ] Глобальная активность (A) выключает всё независимо от календаря; календарь при этом сохраняется.
- [ ] Платформенная блокировка (C) перекрывает A и B; гид не может снять её сам.
- [ ] RLS: гид видит/меняет только свои блоки; турист не читает блоки напрямую.
- [ ] Все изменения доступности прослеживаются в аудите (кто/когда/источник).
- [ ] Туристу нигде не раскрывается причина недоступности.

## Короткий вывод

Не заменяем кнопку, а строим связку трёх уровней:

- **кнопка (A)** — общий режим активности гида;
- **календарь (B)** — расписание исключений по датам/окнам;
- **блокировка платформы (C)** — перекрывает всё, управляется поддержкой;
- **серверная резолюция** — единый источник правды для всех бронирований и именных запросов.

Так функция повторяет привычную модель Tripster, но не ломает уже внедрённый переключатель активности.
