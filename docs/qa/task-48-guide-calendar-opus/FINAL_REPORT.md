## 1. Что реализовано

Второй слой доступности гида (**B — календарь недоступности**) поверх уже
существующих слоёв **A** (`is_available`, главный рубильник) и **C**
(`account_status`, блокировка платформой). Слой A и C не изменены и не сломаны.

Гид может закрыть:

- **целый день** (`kind: "day"`);
- **диапазон дат** — отпуск с… по… (`kind: "range"`);
- **окно времени в конкретный день** — напр. 14:00–18:00 (`kind: "window"`);

плюс указать **приватную причину** (не видна туристу) и **удалить** период
(мягкое удаление — строка остаётся для аудита).

### Единая точка запрета (chokepoint)

Провoдник работает по **бид-модели**, а не по мгновенному бронированию.
Заявка туриста (`traveler_requests`) не содержит `guide_id` —
`preferred_guide_slug` это только метаданные (см. `EXECUTION_NOTES.md`).
Единственное место, где гид привязывает себя к **свободно выбранному** времени —
**создание предложения** (`guide_offers.starts_at/ends_at`). Всё, что ниже по
цепочке, наследует уже проверенное время:

- `counter_offer` копирует `starts_at` из исходного предложения;
- `accept_offer` и `createBooking` копируют время предложения в бронь.

Поэтому слой B ставится **в одной точке — на вставку предложения**:

1. **RLS** — политика `guide_offers_insert` дополнена конъюнктом
   `NOT public.guide_interval_blocked(guide_id, starts_at, ends_at)`. Это
   закрывает обход через прямой вызов API/БД.
2. **Серверный экшен** — `submitOfferAction`/`editOfferAction` делают ту же
   проверку до вставки и возвращают понятное сообщение
   («Этот период закрыт в вашем календаре…»).

Гарантия: **ни через UI, ни напрямую через API нельзя создать бронь на закрытый
интервал**, потому что любая бронь порождается принятым предложением, а
предложение на закрытый слот не создаётся.

### Правила резолюции (реализованы)

- пересечение половинооткрытых интервалов `[start, end)` — union-семантика:
  хватает одного пересечения;
- `all_day`/диапазон нормализуются к границам суток по МСК (UTC+3);
- пустые границы (предложение без времени) → не блокируется;
- существующие отклики/брони закрытием **не отменяются** — при создании блока,
  пересекающего активные предложения гида, возвращается предупреждение.

### Приватность и аудит

- турист никогда не читает таблицу блоков напрямую и не видит причину;
- RLS: гид видит/меняет только свои блоки; админ — все;
- поля аудита: `created_by`, `source` (`manual|calendar|system`),
  `created_at`, `updated_at` (триггер), `deleted_at` (мягкое удаление).

## 4. Доказательства Context7

- **Supabase** (`/supabase/supabase`):
  - RLS-паттерн «владелец управляет своими строками» через
    `(select auth.uid()) = user_id`, отдельные политики на select/insert/update/
    delete — использован для `guide_availability_blocks`.
  - Range/overlap: оператор `&&` и половинооткрытые интервалы `[)` —
    сверялся при проектировании резолюции пересечений (в итоге для надёжности
    в SQL использованы простые сравнения `start < p_end AND p_start < end`,
    эквивалентные `&&`, без риска на вырожденных границах).
- **Next.js** (`/vercel/next.js`, v16.2.9): Server Actions + `revalidatePath`
  после мутации, серверная zod-валидация через `safeParse` — применено в
  `availabilityBlocks.ts` (revalidate `/guide/profile`).

## 5. Известные ограничения и follow-up

1. **`database.types.ts` не перегенерирован.** Локальная docker-БД содержит
   несвязанный drift другого воркспейса (`_bak_*` таблицы, смена
   `PostgrestVersion`), поэтому полный `bun run types` притащил бы ~465 чужих
   строк. Клиент Supabase в проекте не типизирован `<Database>` (см.
   `server.ts`), поэтому `.from("guide_availability_blocks")` компилируется без
   записи в типах — тот же приём уже применён для `guide_availability_events`
   (его тоже нет в committed types). Follow-up для Hermes: перегенерировать типы
   на чистой БД и добавить `guide_availability_blocks` + функцию.
2. **UX-предупреждение именного запроса (план §3.2)** — осознанно вне MVP.
   Именной запрос не резервирует гида и может уйти другим гидам; жёсткий запрет
   на его создание — продуктовый вопрос из §10 плана. Жёсткая гарантия
   безопасности обеспечена на chokepoint предложения. Детали — в
   `EXECUTION_NOTES.md`.
3. **Живой браузерный проход не выполнен** (Ревизия Бека, 1280/375px). В
   локальной БД нет сид-данных одобренного гида для авторизованной сессии, а
   деплой/пуш запрещены. Проверено через typecheck/lint/test/build + SQL-пруф
   RLS. Рендер и форматирование покрыты юнит-тестами компонента. Follow-up для
   Hermes: пройти `/guide/profile` под одобренным гидом на превью.
4. **Pre-commit hook.** Общий hook (`/Users/idev/provodnik/.git/hooks/`)
   ссылается на скрипты `lint:canon`/`lint:dead`, которых нет в `package.json`
   этой ветки (drift hook↔ветка). Полный гейт прогнан вручную (см. п.3), коммит
   сделан с `--no-verify`. Это не обход проверок — это обход сломанного hook.
5. **Богатый календарь-грид (мини-календарь + таймлайн 30 мин, план §4.1)** —
   Этап 3. В MVP сделан честный рабочий раздел «Закрытые периоды» (форма + список
   с удалением). Существующий per-listing календарь (`listing_schedule_extras`) —
   отдельная сущность, не тронут.
