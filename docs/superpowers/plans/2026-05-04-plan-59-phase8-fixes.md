## Task 59-1 [P0] — Add `channel`, `status`, `read_at` columns to notifications table

**Source:** T058-F001 + T058-F002
**Type:** MIGRATION — Supabase MCP (orchestrator executes directly)

**Root cause:** `NotificationBell.tsx` queries:
```
.eq("channel", "inbox")
.neq("status", "read")
```
and updates:
```
.update({ status: "read", read_at: new Date().toISOString() })
```
But the `notifications` table (created in `20260401000001_schema.sql`) only has:
`id, user_id, kind, title, body, href, is_read, created_at`

No `channel`, `status`, or `read_at` columns → HTTP 400 on every load.
The `NotificationRow2` TypeScript type (types.ts:516) correctly describes the INTENDED schema;
the DB migration was never updated to match.

**Fix — run via Supabase MCP (project `yjzpshutgmhxizosbeef`):**

Step 1 — apply migration:
```sql
-- Add missing columns to notifications
ALTER TABLE public.notifications
  ADD COLUMN IF NOT EXISTS channel TEXT NOT NULL DEFAULT 'inbox',
  ADD COLUMN IF NOT EXISTS status  TEXT NOT NULL DEFAULT 'unread',
  ADD COLUMN IF NOT EXISTS read_at TIMESTAMPTZ;

-- Backfill: sync status from is_read boolean
UPDATE public.notifications
   SET status = 'read', read_at = updated_at
 WHERE is_read = true;

-- Index for the NotificationBell query pattern
CREATE INDEX IF NOT EXISTS notifications_channel_status_idx
  ON public.notifications (user_id, channel, status, created_at DESC);
```

Step 2 — write the migration file so it's tracked:
File: `provodnik.app/supabase/migrations/20260504000001_notifications_channel_status.sql`

Step 3 — commit: `fix(notifications): add channel/status/read_at columns to notifications table`

Step 4 — push to origin/main.

**Verify:** Direct Supabase REST call returns 200 (not 400):
`GET .../rest/v1/notifications?select=*&user_id=eq.<guide-uid>&channel=eq.inbox&status=neq.read&limit=1`

## Task 59-3 [P1] — Fix "История" chip filter — update seed listing categories

**Source:** T056-F02
**Type:** Admin SQL — orchestrator executes via Supabase MCP (no cursor-agent needed)

**Root cause:** The chip filter at `/listings` maps "История" → exact match on `listing.themes[0]`.
`themes[0]` is built from `format` field which is built from `row.category` (queries.ts:275).
Seed listings have `category = 'Историческая прогулка'` (not `'История'`), so the chip returns 0 results.
The `PublicListingTheme` type only accepts: "Еда" | "История" | "Природа" | "Фотография" | "С семьей" | "Несезон".

**Fix — run via Supabase MCP:**

Step 1 — identify listings with history-themed titles:
```sql
SELECT id, slug, title, category
FROM listings
WHERE title ILIKE '%истори%' OR title ILIKE '%историч%'
   OR category ILIKE '%истори%'
ORDER BY title;
```

Step 2 — update at least 2 history-themed listings to category = 'История':
```sql
UPDATE listings
   SET category = 'История'
 WHERE slug IN (
   'moscow-hidden-history-tour'
   -- add slug of at least one more history-themed listing here
 );
```

Step 3 — verify:
```sql
SELECT slug, title, category FROM listings WHERE category = 'История';
```
Must return ≥ 2 rows.

Step 4 — live check: open https://provodnik.app/listings and click "История" chip — must show ≥ 2 cards.

**No migration file needed** (seed data update, not schema change). No commit.
