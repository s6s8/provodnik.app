# Tripster v1 Full Replication ‚ÄĒ Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking. This plan is the authoritative task list; per-wave cursor-agent prompt files under `.claude/prompts/tripster-v1/` are generated just-in-time at dispatch and wrap a wave's task section with the ¬ß6.3 context blocks (Project CLAUDE.md).

**Goal:** Ship full Tripster replication in Provodnik across 14 phases, 42 waves. All 8 listing types, full guide + traveler surfaces, notifications, reputation, moderation, peripherals. No deferrals.

**Architecture:** Next.js 15 App Router, Supabase Postgres + RLS + pg_cron, PostGIS, Yandex Maps JS, Tremor for charts. One editor shell + `SECTIONS_BY_TYPE` branching over 8 listing types. Two detail templates (excursion-shape, tour-shape) plus transfer. Feature-flag gated via `FEATURE_TRIPSTER_V1` + 12 sub-flags.

**Tech Stack:** TypeScript, Next.js 15, Supabase, PostgreSQL 15, PostGIS, pg_cron, Tailwind, Tremor, Zod, Playwright, Vitest.

**Spec:** `docs/superpowers/specs/2026-04-12-tripster-v1-design.md`

## Execution Handoff

Plan saved to `docs/superpowers/plans/2026-04-12-tripster-v1.md`.

Given the scope ‚ÄĒ 42 waves, ~55 cursor-agent dispatches, ~6 weeks ‚ÄĒ and the Primary Coder rule from Project CLAUDE.md (cursor-agent via dispatch wrapper, native agents as wrapping executors), execution uses the **subagent-driven-development** model:

1. Orchestrator generates per-wave cursor-agent prompt files just-in-time into `.claude/prompts/tripster-v1/` as each phase's preconditions land
2. Each prompt file wraps a wave section from this plan with the ¬ß6.3 context blocks (PROJECT_MAP excerpt, PATTERNS excerpt, ERRORS excerpt, ANTI_PATTERNS excerpt, DECISIONS excerpt, API_REFERENCE excerpt, environment block, done criteria)
3. Native Claude agents dispatch cursor-agent per wave via `node .claude/logs/cursor-dispatch.mjs <prompt-file>`
4. Two-stage review per wave before merge; re-dispatch mutated prompt on failure
5. Phase-level checkpoint tags after every successful phase
6. Post-work protocol runs after every merge
