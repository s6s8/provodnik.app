## Task 58-3 [P1] — Fix notifications API 400 on guide pages

**Source:** T038-FIX-002
**Type:** MIGRATION — Supabase MCP (no cursor-agent needed)

**Root cause:** `GET ...notifications?...channel=eq.inbox...` returns HTTP 400 on every guide page.
The column exists (`channel: "inbox" | "email" | "telegram" | "push"` in types.ts) and the query
is constructed correctly in `NotificationBell.tsx`. The 400 likely means an RLS SELECT policy
is missing for the guide role on the `notifications` table.

**Steps:**
1. Check existing policies:
   ```sql
   SELECT policyname, cmd, qual FROM pg_policies WHERE tablename = 'notifications' ORDER BY cmd;
   ```
2. If no SELECT policy exists for the guide's own rows, create migration
   `provodnik.app/supabase/migrations/20260503000004_notifications_guide_rls.sql`:
   ```sql
   -- Allow authenticated users to read their own notifications
   CREATE POLICY "user_select_own_notifications"
     ON notifications FOR SELECT
     TO authenticated
     USING (user_id = auth.uid());
   ```
3. Apply via Supabase Management API (project `yjzpshutgmhxizosbeef`).
4. Commit the migration file: `fix(notifications): add user SELECT RLS policy`
5. Push to origin/main.

**Verify:** `bun run typecheck` exits 0; `GET ...notifications?channel=eq.inbox` returns 200 (not 400).
